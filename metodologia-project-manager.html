Metodología interna · Gestión de Cartera

Cómo gestionamos los proyectos

El sistema que mantiene cada proyecto vivo, documentado y al día. Una sola forma de trabajar, un solo schema, cero proyectos perdidos en el limbo.

Versión 2.0 · schema v2 Actualizado 2026-06-04 Para uso compartido — Guille + Facu

1 Qué es esto

El motor de gestión de toda la cartera de proyectos. Mantiene cada proyecto alineado a un mismo estándar, ejecuta la apertura completa de un proyecto nuevo sin idas y vueltas, y se actualiza solo cuando hay actividad.

El principio: un proyecto no es una carpeta suelta. Es cuatro cosas a la vez, siempre, sin excepción:

📁 Ficha (README)
+
🗂️ Carpeta wiki
+
💻 Carpeta en disco
+
🐙 Repo GitHub

No importa si el proyecto aún no arrancó o si nunca va a tener código. La estructura existe igual, porque es el sistema de seguimiento. Y la ficha (el frontmatter) es la base de datos: si falta un campo, el seguimiento está roto.

Regla de oro. Se pregunta solo lo que no se puede deducir del contexto, todo junto en un único bloque al inicio. Después se ejecuta todo de corrido y se reporta al final. Nada de preguntar de a una cosa.

2 El schema estándar

21 campos organizados en 6 bloques. Esto es lo que define a cada proyecto. Todos los campos van siempre presentes en la ficha — si no aplica, va vacío (""), nunca ausente.

🪪 Identidad

CampoObligatorioQué es
nombreSÍTítulo legible del proyecto.
areaSÍauckland · asesorias · apps · revolucia-labs · personal
clienteopc.Cliente externo si aplica. Vacío = interno.

👥 Responsabilidad (RACI)

CampoObligatorioQué es
stakeholderSÍDe quién es el proyecto. darío (cartera Auckland) o personal (Guille).
responsableSÍQuién ejecuta / hace de PM. Default guille. Ampliable al equipo.

🚦 Estado / Salud

CampoObligatorioQué es
estadoSÍactivo · pausado · completado · por-lanzar · critico · archivado · evaluando
prioridadSÍalta · media · baja
saludSÍSemáforo RAG: en-tiempo en-riesgo atrasado
progresoSÍAvance 0-100. completado = 100.

📅 Tiempo

CampoObligatorioQué es
fecha_creacionSÍCuándo nació en el sistema. No cambia.
fecha_inicioopc.Cuándo arrancó la ejecución real. Vacío = todavía no arrancó (backlog).
fecha_objetivoopc.Deadline de cierre. Vacío = sin fecha definida.
proxima_accionSÍEl siguiente paso concreto. Si está activo, no puede estar vacío.
fecha_proxima_accionopc.Si ya pasó y el proyecto está activo → está vencida.
ultima_actualizacionSÍSe actualiza en cada toque. +14 días en activo → stale.

📝 Contenido

CampoObligatorioQué es
objetivoSÍDefinition-of-done en 1 frase: qué significa "terminado".
tagsopc.Etiquetas para filtrar. Se derivan de área + nombre + cliente.

🔗 Integración

CampoObligatorioQué es
urlopc.URL viva/deploy (app publicada, landing, dominio).
repoopc.Repositorio de código en GitHub.
carpeta_discoopc.Ruta en disco (ej: C:\Proyectos\X).
notion_idopc.Puente de sync con Notion. Vacío = aún no sincronizado.

3 Las cuatro operaciones

Todo lo que hace el sistema cae en una de estas cuatro cosas.

Operación 1

Nuevo

Abrir un proyecto de cero. Naming → preguntar lo mínimo en un bloque → crear las 4 patas → reportar. Sin interrupciones.

Operación 2

Validar

Auditar fichas existentes: detectar campos faltantes, fechas vencidas, proyectos olvidados. Completar lo que se puede y preguntar el resto.

Operación 3

Seguimiento

Actualizar un proyecto cuando hay actividad real (una reunión, un avance, un mail). Recalcula salud, progreso y próxima acción.

Operación 4

Sync Notion

Espejar las fichas a Notion. Preparado pero todavía no activo — la wiki es la fuente, Notion sería el espejo.

Operación 1 · Crear un proyecto nuevo

Naming
→
Preguntar (1 bloque)
→
Ejecutar todo
→
Reportar

Naming primero. El mismo slug en los cuatro sistemas. Siempre lowercase-con-hyphens, sin acentos, corto y descriptivo, con el año al final si es relevante.

# El mismo nombre en todos lados
wiki/proyectos/[slug]/README.md   ← ficha PM
C:\Proyectos\[slug]\              ← carpeta física
github.com/Gcramirez/[slug]       ← repo GitHub
Google Drive / [área] / [slug]     ← carpeta Drive

Se pregunta solo lo que no se puede inferir, todo en un bloque: stakeholder, objetivo, área (si no es obvia), prioridad, fecha objetivo, fecha de inicio, próxima acción + fecha, cliente y si el repo ya existe con otro nombre.

Lo demás se autocompleta sin preguntar:

fecha_creacionhoy

saluden-tiempo
progreso0
responsableguille

Después ejecuta todo de corrido: crea la carpeta wiki, escribe el README desde el template, crea la carpeta en disco, crea el repo en GitHub (privado por default, público si es un producto/app), lo registra en el log y, si está en evaluando, lo suma al backlog. Recién al final reporta lo hecho.

Operación 2 · Validar y completar fichas

Se dispara con cosas como "validá la ficha de X", "revisá toda la cartera" o "qué proyectos les falta info". Lee el frontmatter de cada README y corre las reglas de validación, reportando por severidad:

  • CRÍTICO  Proyecto activo sin próxima acción · fecha de próxima acción vencida
  • WARNING  Stale (+14 días activo sin tocar) · pausado +30 días · salud incoherente con las fechas
  • INFO  Campos obligatorios vacíos · completado con progreso ≠ 100
  • AZUL  Sin carpeta de disco · sin repo

Lo que se puede derivar se autocompleta; lo que requiere criterio se pregunta en un único bloque. En cada ficha tocada se refresca la ultima_actualizacion a hoy.

Operación 3 · Seguimiento

Cuando hay actividad real sobre un proyecto, se actualiza sin preguntar: la próxima acción y su fecha, el estado si cambió, el progreso, la última actualización y una línea nueva en el log del README (| hoy | qué cambió · por qué |).

El semáforo de salud se recalcula así:

SituaciónSalud
Avanzando, deadline con margenen-tiempo
Sin avance +7 días, o deadline a menos de 21 días sin progresoen-riesgo
Próxima acción vencida, o fecha objetivo ya pasadaatrasado

Al completar: estado completado, progreso 100, fecha de cierre en el log, se saca del backlog.
Al pausar: estado pausado, próxima acción "Revisar en triaje mensual", y se suma a la lista de pausados para el triaje del día 19 de cada mes. Un proyecto pausado sin acción definida muere en el limbo — por eso el triaje mensual decide: retomar / archivar / mantener / delegar / esperar.

4 Las reglas que no se rompen

  • Siempre las cuatro patas. Carpeta wiki + carpeta disco + repo + README completo. Sin excepción.
  • Ejecutar sin consultas intermedias. Un bloque de preguntas al inicio, después se ejecuta todo, se reporta al final.
  • Todos los campos presentes. Vacíos van como "", nunca ausentes.
  • Stakeholder siempre claro. Si no es obvio, se pregunta.
  • Las fechas se validan contra el sistema. Nunca se precalcula "hoy" de memoria.
  • No inventar. Se proponen defaults; lo que requiere criterio se pregunta.
  • El schema es uno solo. Cualquier cambio se hace en schema.yaml y se registra como decisión.
  • No se borra nada. Los proyectos muertos pasan a estado: archivado.
  • Todo lo que está en "evaluando" va al backlog. No se pierde de vista.

5 Cómo se dispara cada cosa

No hace falta memorizar comandos. Se le habla en lenguaje natural y el sistema mapea a la operación correcta.

Lo que decísLo que hace
"creá / abrí / anotá el proyecto …"NUEVO
Llega un mail / reunión con un proyecto nuevoNUEVO (automático)
"validá / completá la ficha de …"VALIDAR (uno)
"revisá / barré toda la cartera"VALIDAR (todos)
"estandarizá los proyectos"VALIDAR (todos, migración v2)
"avanzó …", "actualizá el seguimiento de …"SEGUIMIENTO
Se procesa una reunión / devlog / emailSEGUIMIENTO (automático)
"sincronizá con Notion"SYNC (cuando se active)
La base de PM detrás de esto. El schema está armado sobre estándares reconocidos: RACI (quién es sponsor vs. quién ejecuta), semáforo RAG de salud (verde/amarillo/rojo), definition-of-done explícita por proyecto, y próxima acción siempre viva. Nada de invento — es PM clásico aplicado a una cartera de muchos proyectos en paralelo.
